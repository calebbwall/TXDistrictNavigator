C) Jobs + scheduling:
Create new job modules under server/jobs/:
1) pollRssFeeds.ts
- For each enabled rss_feed:
  - Fetch using conditional headers:
    - If rss_feeds.etag exists, send If-None-Match
    - If rss_feeds.last_modified exists, send If-Modified-Since
  - If 304 Not Modified: update last_polled_at and continue
  - If 200: parse RSS XML (use a small library already in deps or add a lightweight parser)
  - Upsert rss_items by (feed_id + guid). Insert only new items.
  - For each new item:
    - Create an alert row for user "default"
    - Trigger targeted refresh by enqueuing task in a lightweight in-process queue:
      - Parse item.link to determine committee/hearing/bill identifiers if possible
      - Call targeted refresh functions (see below)

2) refreshDailyLegislative.ts
- Runs daily at 5:00 AM America/Chicago
- Steps:
  - Determine scoped committees:
    - committees of saved officials + committees in user_subscriptions (type=COMMITTEE)
    - If no saved officials exist yet, fallback to a small safe default set (e.g., top N committees already in db) to avoid empty UI
  - For each scoped committee:
    - refresh upcoming hearings list
    - for new/changed hearings, fetch detail pages to fill hearing_details + agenda + witness list
  - Refresh session calendars (House/Senate) into legislative_events (event_type=FLOOR_CALENDAR/SESSION_DAY) using fingerprints to avoid churn
  - Update last_seen_at for items found; optionally mark stale/cancelled if previously existed but disappeared

3) targetedRefresh.ts (utility module)
Export functions used by pollRssFeeds and daily refresh:
- refreshCommitteeHearings(committeeId, windowDays=14)
- refreshHearingDetail(externalId or sourceUrl)
- refreshBillHistory(billNumber) but ONLY for bills that are:
  a) on a hearing agenda, or
  b) explicitly subscribed (type=BILL)
