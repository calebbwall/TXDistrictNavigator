You are working in my existing Replit project “TXDistrictNavigator”. Implement the “Committees” feature under Profile → Tools, with weekly refresh + change detection, and integrate committees onto each official’s PUBLIC card.

========================================================
FEATURE OVERVIEW (Acceptance criteria)
========================================================

A) Profile → Tools → Committees
1) In the Profile tab under “Tools”, add a new tool item: “Committees”.
2) Inside Committees:
   - Let the user choose either:
     - “Texas House Committees”
     - “Texas Senate Committees”
3) In each chamber section:
   - Show a list of all committees for that chamber
   - Selecting a committee shows:
     - Committee name
     - A list of committee members with their role/title (Chair, Vice Chair, Member, etc.)
4) Selecting a committee member navigates to that official’s card (same as normal official navigation).

B) Official card PUBLIC tab: Committees section
5) On each official’s PUBLIC tab, add a “Committees” section listing:
   - Committee Name — Role
   (one line per committee membership)
6) The committees displayed must match the committee membership data ingested from the sources.

C) Weekly refresh with change detection
7) Committees data should be checked weekly early Monday morning (Central Time), using the SAME schedule and fingerprint patterns used for Officials/GeoJSON:
   - If unchanged, do nothing
   - If changed, refresh and store updated committee + membership data
8) Add an admin endpoint to force refresh:
   - `POST /admin/refresh/committees`
   - Protected by `x-admin-token` = `ADMIN_REFRESH_TOKEN`
   - Response indicates checked datasets, changed vs unchanged, refreshed yes/no

========================================================
DATABASE SCHEMA (Drizzle)
========================================================

Create new tables (or equivalent) for:
1) `committees`
   - committeeId (PK)
   - chamber (TX_HOUSE or TX_SENATE)
   - name
   - slug or externalId (unique per chamber)
   - sourceUrl
   - updatedAt
2) `committee_memberships`
   - membershipId (PK)
   - committeeId (FK)
   - officialPublicId (FK to existing officials table) OR (source + sourceMemberId) mapping fields
   - roleTitle (text)
   - sortOrder (optional)
   - updatedAt

Mapping requirement:
- Committee members must be linkable to the existing official records.
- Prefer matching by:
  - chamber/source + district number or member id if available
  - or by normalized name matching as fallback (but keep it deterministic and logged)
- Do NOT create duplicate officials. Just link to existing officials.
