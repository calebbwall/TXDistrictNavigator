Replit Continuous Agent Prompt

Feature Set: Private Notes & Engagement Tracking + About Screen (No PIN, No Removal of Existing Text)

Objective

Extend the existing app codebase to add:
	1.	Private “Notes & Prayer” per official (multi-entry, timestamped, follow-up flag).
	2.	“Last Engaged” tracking per official with optional notes.
	3.	An About screen with purpose text and a link to https://www.capitolcommissiontexas.org.

⚠️ Do NOT remove or alter any existing custom text, including the “Made for the State Minister of Texas” message.

All new data must remain private, device-local, and must never be overwritten by refreshes.

⸻

1) Private Notes & Prayer

Data model (private, per official)

Store per official (keyed by {source, districtNumber} to survive vacancies and refreshes):

notesPrayerEntries: Array<{
  id: string;
  createdAt: string; // ISO timestamp
  text: string;
  followUpNeeded: boolean;
}>

Requirements
	•	Multiple entries, newest first
	•	Display date, text, and follow-up badge
	•	Add entry UI:
	•	Multiline text input
	•	Follow-up checkbox/toggle
	•	Save button
	•	Delete entry is required
	•	Edit entry optional (only if trivial)

UI placement

In OfficialProfileScreen.tsx:
	•	Section title: “Private Notes & Prayer”
	•	Subtext: “Private to this device.”
	•	Scrollable list of entries
	•	“Add Note” action

⸻

2) Last Engaged Tracker

Data model (private, per official)

engagementLog: Array<{
  id: string;
  engagedAt: string; // ISO timestamp
  summary?: string;  // optional notes
}>

Requirements
	•	Display “Last engaged: ” (derived from latest entry)
	•	Add entry UI:
	•	Date/time picker (default = now)
	•	Optional summary field
	•	Save button
	•	List previous engagement entries (newest first)
	•	Delete entry required

UI placement

Below Notes & Prayer in OfficialProfileScreen

3) Storage & Persistence Rules (Critical)
	•	Store Notes & Prayer + Engagement Log in private local storage (AsyncStorage or existing storage layer).
	•	Public data refresh jobs must never overwrite private storage.
	•	Notes must persist even if:
	•	an official changes
	•	a seat becomes vacant
	•	data is refreshed weekly
	•	Key private data using:

private:{source}:{districtNumber}

	•	vacancies
	•	data-source churn
…and you avoid “Austin problem” regressions you’ve already run into.
