TASK 2 — Diagnose + fix Android “first tap returns nothing” bug
========================================================

Symptom:
- On Android, sometimes the first tap inside a district produces no results (no district selection / no card).
- A second tap in the same place works and returns data.

Goal / Acceptance criteria:
1) Tap-to-search works reliably on Android on the first tap.
2) No regressions on iOS.
3) No accidental interference with pan/zoom, draw tool, or overlays.

Likely causes to investigate (do this methodically):
A) Tap event not firing consistently
- MapView `onPress` handler may not fire due to:
  - overlay views intercepting touches (`pointerEvents`)
  - press gestures being swallowed by a parent container
  - incorrect `onPress` wiring depending on active tool state
B) Race condition in state
- Selection depends on overlay GeoJSON loading or being in memory.
- On Android, first tap might happen before geojson or spatial index is ready.
C) Coordinate order / precision issues
- Lat/lng vs lng/lat confusion, or rounding issues in hit-test.
D) Debounce / throttling
- A stale state or early return could be preventing the first selection.
E) Concurrent tool modes
- Tap-to-search may be ignored when draw tool is active or just exited.

Implementation requirements for diagnosis:
1) Add temporary, minimal debug logs (guarded so they can be turned off):
   - Log when MapView onPress fires (timestamp, coordinate)
   - Log which tool mode is active at press time
   - Log whether overlays are loaded + counts of features for each overlay
   - Log hit-test result (district found or not)
2) Reproduce logic:
   - If no overlays loaded yet, do not attempt hit-test; show a brief “Loading overlays…” and retry automatically once loaded.
3) Fix the root cause:
   - Ensure overlays/features are ready before enabling tap-to-search
   - Ensure pointerEvents do not block taps on Android
   - Ensure tap-to-search and draw-to-search modes are mutually exclusive
   - Ensure hit-test uses correct coordinate format

Hard requirement:
- After debugging, remove or gate logs behind a `DEBUG_MAP_TAPS` constant/env flag so production logs aren’t spammy.
